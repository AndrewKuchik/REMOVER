### Запись 3 — Фаза 2: качество края (деконтаминация)
**Дата:** 2026-07-03
**Задача:** Убрать цветную каёмку и смягчить край автоматически, с ползунками.
**Сделано:** ✅ Да.
- Новый конвейер: берём чистую маску (`segmentForeground`) и сами накладываем на оригинал в полном разрешении.
- Модуль `refine.js`: эрозия (обрезка каёмки) + деконтаминация (затекание цвета объекта наружу) + feather (мягкость). Ползунки пересчитывают край мгновенно, без повторного прогона нейросети.
- **Проверка в браузере:** тест поймал реальный баг — первая версия деконтаминации убирала лишь 3% каёмки (брала цвет-источник у самого края, а он загрязнён). Исправлено: источник цвета берём глубже края. Результат — жёлтых пикселей в крае **1561 → 0**.
**Оценка эффективности:** 95% — цель достигнута и подтверждена тестом; баг найден и устранён самопроверкой. Минус 5%: деконтаминация на очень больших фото (12+ Мп) может считаться заметное время — оптимизация при необходимости позже.

### Запись 5 — Фаза 4: про-режим BiRefNet
**Дата:** 2026-07-03
**Задача:** Добавить качественный режим (BiRefNet, WebGPU) для сложных кадров, с переключателем.
**Сделано:** ✅ Да (с оговоркой по среде проверки).
- Модуль `engine-pro.js`: BiRefNet_lite через transformers.js, WebGPU с откатом на WASM; маска берётся из альфы и идёт в тот же слой обработки края.
- transformers.js грузится **лениво** (только при выборе «Качество») — быстрый режим остаётся лёгким.
- Переключатель «⚡ Быстро / 💎 Качество»; смена режима переобрабатывает картинку. Предупреждение, если нет WebGPU.
- **Найдено и исправлено при проверке:** запрашивал формат `q8`, которого у модели нет (есть только fp32/fp16) → `constructSessions` падал. Исправлено: WebGPU→fp16, WASM→fp32 + откат.
- **Проверка:** полный инференс в headless невозможен (нет видеокарты, CPU-WASM слишком медленный). Поэтому проверил модель напрямую в Node (нативный runtime): выход RGBA, альфа центра=254, угла=0 — объект сегментирован верно, формат совпадает с ожиданиями браузерного кода.
**Оценка эффективности:** 85% — код корректен и модель проверена; поймал и починил реальный баг формата. Минус 15%: WebGPU-скорость и качество на реальных «волосатых» фото не проверены в этой среде — нужна проверка пользователем на его машине.

### Запись 7 — Фикс по фидбеку: краш «Качества» + прогресс
**Дата:** 2026-07-03
**Задача:** По отзыву пользователя: режим «Качество» падал (`std::bad_alloc` в OrtRun); прогресс завис на 100%; окно загрузки вводило в заблуждение.
**Сделано:** ✅ Да.
- **Причина краха (подтверждена разведкой):** у BiRefNet_lite нет квантованной версии → без WebGPU грузилась fp32 (214 МБ) на CPU/WASM → переполнение памяти. RMBG-1.4 через pipeline не поддерживается в этой версии (проверил — не стал городить хрупкий путь).
- **Решение:** «Качество» = BiRefNet fp16 на WebGPU; без WebGPU — надёжный откат на @imgly (не падает по памяти). Вход уменьшается до 2048px. Нормальная проверка WebGPU через `requestAdapter()`.
- **Прогресс:** разделён на «Скачиваю модель X%» и «Обрабатываю… N с» (живой счётчик — не выглядит зависшим). Окно загрузки гарантированно закрывается по готовности.
- **Ошибки:** вместо alert — понятная панель с кнопками «Попробовать снова» / «Переключиться на Быстро».
- **Индикатор:** показывается, какой движок и устройство реально сработали.
- **Проверка в браузере:** режим «Качество» без WebGPU больше не падает — откат на @imgly даёт результат (opaque 44832), alert не выскакивает, подпись «ISNet на процессоре». Быстрый режим работает.
**Оценка эффективности:** 90% — краш устранён, UX прогресса/ошибок исправлен, всё проверено. Минус 10%: путь BiRefNet-на-WebGPU нельзя проверить в headless (нет видеокарты) — нужна проверка пользователем на его машине.

### Запись 8 — Переход на единый движок RMBG + спокойный UX
**Дата:** 2026-07-03
**Задача:** По отзыву: «Качество» зависало/падало, пугающий треугольник ошибки, «бесконечная загрузка». Нужен спокойный, логичный, надёжный UI и максимально доступное качество бесплатно.
**Сделано:** ✅ Да.
- **Диагностика живого сайта:** на свежей версии ошибки при заходе НЕТ (был закэшированный старый вариант). «Бесконечная загрузка» = путь BiRefNet (109 МБ + WebGPU).
- **Разведка:** бесплатных серверных API без ловушек нет; смена хостинга не даёт бесплатного инференса; лучший путь — маленькая модель в браузере + спокойный UX.
- **Решение:** единый движок **RMBG-1.4 q8 (~44 МБ)** через `AutoModel(model_type:'custom')` на WASM. Убраны режимы, @imgly и BiRefNet. Эмпирически проверил, что RMBG грузится в нашей версии (обход ошибки Segformer).
- **UX переделан:** спокойная загрузка (полоса + МБ + «один раз»), обработка со счётчиком, мягкое уведомление вместо треугольника.
- **Проверка в браузере (прод-сборка):** заход без паники; скачивание 44 МБ + вырезание за ~11 с; объект вырезан, разрешение цело.
- Флаг: RMBG-1.4 — лицензия CC BY-NC (некоммерческая). Для прототипа ок, для бизнеса — обсудить.
**Оценка эффективности:** 92% — краш устранён совсем, UX спокойный, качество выше прежнего, всё проверено. Минус 8%: лицензия ограничивает коммерцию; скорость на слабом CPU — несколько секунд (не «мгновенно»), WebGPU-ускорение отложено.

- **Многоагентное исследование** (5 агентов): конкуренты, модели, техники очистки, хостинг. Выводы применены.
- **Очистка маски** (`cleanup.js`): медиана → порог альфы (smoothstep 0.12/0.88) → удаление мелких «островков» (connected components) → морфология open/close. На РОДНОМ разрешении до растягивания. Юнит-тест: реалистичный шум фона 50163→0, объект цел.
- **UX обработки переделан** (паттерн Photoroom): фото сразу показывается размытым на месте результата, спиннер — с задержкой 800 мс (если быстро — без мельканий), чёткие этапы «Настройка (один раз) %» vs «Убираю фон… N с». Реакция кнопок усилена. Приватность вынесена к зоне загрузки.

**Оценка эффективности:** 88% — критбаг убит, грязь чистится, UX спокойный и логичный. Минус 12%: RMBG остаётся мягкой моделью (полностью «в ноль» чистит не всегда) — для топ-краёв нужен BiRefNet (224 МБ, WebGPU) как опция; переезд на Cloudflare (быстрее CPU + приватный репозиторий) — отдельным шагом. Оба предложены пользователю.

### Запись 10 — Прорыв: мгновенный алгоритмический кейер + ленивая нейросеть
**Дата:** 2026-07-03
**Задача:** По запросу — глубокое исследование и «другой путь»: максимум качества без долгих загрузок, автоматизация ручного метода Photopea, отличие от конкурентов.
**Сделано:** ✅ Да (ядро P0+P1).
- **Многоагентное исследование** (9 агентов: 6 разведка + синтез + адверсариал-критик + финал). Вывод: конкуренты все на фото-нейросетях, которые плохи на рисунках/логотипах/сканах — наша ниша. Позиционирование: «мгновенный приватный удалитель для иллюстраций».
- **Мгновенный кейер** (`keyer.js`): автоматизация Photopea-метода (волшебная палочка + Expand/Feather). Правки критика внедрены: эталон фона по полосе краёв с отбраковкой контаминации (объект у края), запечатывание разрывов штрихов (dilate) против «протекания» заливки. Юнит-тесты: базовый/контаминация/кольцо-с-разрывом — 3/3.
- **Ленивая нейросеть** (`router.js`): transformers.js вынесен из стартового бандла (first-paint 515 КБ → **14 КБ**), RMBG грузится только по кнопке «🧠 Обработать нейросетью».
- **Ползунок «Чувствительность фона»** — живой пере-расчёт по прокси (то самое «помочь различить»).
- **Проверка в браузере:** заход мгновенный; рез иллюстрации за **239 мс**, углы чистые (0 точек!), transformers НЕ грузился; чувствительность пересчитывает; ленивая нейросеть по кнопке работает.
**Оценка эффективности:** 90% — прорывное ядро внедрено и проверено, «долгих загрузок» на дефолтном пути больше нет, качество на иллюстрациях чистое. Минус 10%: ещё не сделаны P2/P3 (пред-обработка контрастом, маска по яркости для кружева, ручная кисть для идеала) — следующие заходы.
